# 프론트엔드 테스트 평가서

## 1. 검수 개요

- 검수 대상: Workspace 전체 프론트엔드 (`templates/`, `static/js/`, `static/css/`, Tailwind 템플릿 기반 화면)
- 검수 목적: 사용자 주요 플로우 중심으로 장애 가능성, 사용자 영향도, 복잡도 높은 로직 우선 점검
- 코드 수정 여부: 수정하지 않음
- 주요 확인 기능:
  - 검색/필터/페이지네이션
  - 챗봇 메시지 송수신
  - 상품 상세 찜하기/마이페이지 찜 해제
  - 로그인/비밀번호 찾기 패널 전환
- 우선 검토 기준:
  - 사용자 주요 플로우 직접 영향
  - API 요청/응답 처리, 에러/로딩 처리
  - 이벤트/비동기 중복 실행 가능성
  - 입력 검증, 보안 취약점 가능성
  - 유지보수성(책임 분리, 중복 코드, 하드코딩)

## 2. 주요 로직 분석

### 2.1 우선 검토 대상 선정 이유

| 우선순위 | 검토 대상 | 선정 이유 | 사용자 영향도 |
|---|---|---|---|
| 1 | 챗봇 메시지 렌더링/전송 | 사용자 입력 + 서버 응답을 HTML로 렌더링하고 API 호출 빈도가 높아 보안/장애 영향 큼 | High |
| 2 | 찜하기/찜해제 API 연동 | 클릭 이벤트 기반 상태 변경이며 중복 클릭/실패 처리 미흡 시 데이터 신뢰도 저하 | High |
| 3 | 검색 필터/페이지네이션 | 트래픽과 사용 빈도가 높고 파라미터/입력 검증 복잡도가 큼 | Medium |

### 2.2 관련 파일 및 함수

| 구분 | 파일 경로 | 함수/컴포넌트명 | 역할 |
|---|---|---|---|
| API 연동 | `static/js/chatpage.js` | `sendMessage` | 챗봇 POST 요청, 응답 반영, 로딩 상태 제어 |
| API 연동 | `templates/components/product/product_actions.html` | `productPageWishlistToggle` | 상품 상세 찜 토글 API 호출 |
| API 연동 | `templates/components/account/recent_products.html` | `mypageWishlistToggle` | 마이페이지 찜 해제 API 호출 |
| 상태 관리 | `static/js/chatpage.js` | `messages`, `inFlight`, `setBusy` | 메시지 상태 및 중복 요청 방지 |
| 이벤트 처리 | `static/js/search/filter.js` | `initSearchFilter`, `renderChipGroups`, `form submit handler` | 필터 이벤트, 파라미터 동기화 |
| 이벤트 처리 | `static/js/search/pagination.js` | `initPagination` | 페이지네이션 렌더링 및 URL 파라미터 생성 |
| UI 렌더링 | `templates/components/search/*` | 필터/결과/페이지네이션 템플릿 | 검색 결과/빈 상태 UI |
| 유틸 함수 | `static/js/search/filter.js` | `clampRangeInput`, `formatRangeValue` | 범위 입력값 정규화/보정 |

### 2.3 현재 동작 흐름

1. 검색 페이지는 `filter.js` + `pagination.js` + `searchpage.js`로 초기화되며, URL 파라미터 기반으로 필터 UI를 복원하고 제출 시 빈 파라미터를 제거해 요청합니다.
2. 챗봇 페이지는 `chatpage.js`에서 메시지 목록 상태를 관리하고, 전송 중(`inFlight`)에는 입력/버튼 비활성화 및 임시 로딩 메시지를 표시합니다.
3. 상품 상세/마이페이지의 찜 관련 동작은 템플릿 인라인 스크립트에서 `fetch`로 서버와 통신하고 성공 시 DOM을 갱신합니다.

### 2.4 발견된 문제점

| 심각도 | 문제 | 원인 | 영향 |
|---|---|---|---|
| High | 챗봇 마크다운 링크/이미지 URL 스킴 검증 부재 | `renderMarkdown`에서 `[text](url)`, `![alt](url)`를 HTML로 치환 시 `javascript:`/`data:` URL 차단 없음 | 악성 링크 노출/클릭 유도 가능성, 보안 리스크 |
| Medium | 찜 토글 연속 클릭 시 중복 요청 가능 | `productPageWishlistToggle`, `mypageWishlistToggle`에 in-flight 가드/버튼 disable 없음 | 상태 뒤집힘, 서버 부하, UI/서버 상태 불일치 가능 |
| Medium | 마이페이지 찜해제의 비정상 응답 처리 취약 | `response.json()` 전제 처리(리다이렉트/비JSON 대응 미흡), 실패 시 콘솔 로그 위주 | 사용자는 실패 원인 인지 불가, 동작 불안정 체감 |
| Medium | 상품 상세 찜하기 실패 시 사용자 피드백 부족 | `.catch`에서 `console.error`만 수행 | 네트워크/서버 실패 시 사용자 관점 무반응 |
| Low | 비밀번호 찾기 패널이 UI 전환만 있고 실제 제출 처리 부재 | `reset_password_form` 버튼 `type="button"` + API 연결 없음 | 기능 기대 대비 오작동 인식 가능 |
| Low | TypeScript 부재로 타입 안정성 검증 한계 | JS + 템플릿 기반 구조 | 런타임 오류 사전 차단 한계 (확인 필요) |

### 2.5 수정 제안

> 실제 코드는 수정하지 말고, 어떤 방식으로 수정하면 되는지만 설명해 줘.

- 제안 1: 챗봇 렌더링 전 URL allowlist(`http`, `https`) 검증을 추가하고, 나머지 스킴은 텍스트 처리로 강등.
- 제안 2: 찜하기/찜해제 버튼에 요청 중 disable + 중복 클릭 가드 + 완료 후 복구 패턴을 공통 유틸로 통합.
- 제안 3: `fetch` 공통 응답 처리 유틸을 도입해 `response.ok`, `content-type`, JSON 파싱 실패, 리다이렉트 케이스를 일관 처리하고 사용자 피드백(토스트/문구)을 표준화.

| 모바일 화면 | 작은 화면에서 실행 | UI 깨짐 없음 | 클래스 구성상 반응형 고려됨. 실제 디바이스 렌더링은 확인 필요 | Warning |

## 4. 프론트엔드 코드 품질 검수

## 4.1 성능 최적화

### 발견된 문제

- 찜하기/찜해제 연속 클릭에 대한 요청 디바운스/락 처리 없음
- 인라인 스크립트가 여러 템플릿에 분산되어 공통 처리 재사용이 어려움

### 영향

- 동일 API 중복 호출로 불필요한 네트워크 부하
- 유지보수 시 수정 범위 확대로 회귀 가능성 증가

### 개선 방향

- 버튼 액션 공통 유틸(요청 중 상태/복구/에러 처리)로 통합
- 인라인 스크립트를 페이지별 JS 모듈로 이동해 책임 분리

---

## 4.2 보안 취약점

### 발견된 문제

- 챗봇 마크다운 렌더링 시 링크/이미지 URL 스킴 검증 없음 (`javascript:` 가능성)
- 외부 링크는 `noopener` 처리되어 있으나 URL 자체 유효성/정책 검증 부재

### 영향

- 악성 링크 클릭 유도/보안 사고 가능성
- 사용자 신뢰 하락 및 보안 이슈 대응 비용 증가

### 개선 방향

- URL 파서 기반 allowlist 검증 (`http`, `https`) 적용
- 렌더링 전 Markdown sanitizer 또는 안전 렌더러 도입 검토

---

## 4.3 유지보수성

### 발견된 문제

- 템플릿 내부 인라인 함수(`onclick`, `<script>`) 다수 존재
- API 에러 처리 방식이 페이지별로 불균일(알림/콘솔 로그 혼재)
- TypeScript 미도입으로 타입 기반 회귀 방지 체계 부재

### 영향

- 코드 추적/재사용성 저하
- 예외 케이스 누락 가능성 증가
- 변경 시 런타임 리스크 증가

### 개선 방향

- 프런트 공통 유틸 레이어(`apiClient`, `uiFeedback`) 도입
- 이벤트 바인딩을 JS 모듈에서 통일 관리
- 점진적 TS 도입(핵심 모듈 우선) 또는 JSDoc 타입 강화

---

## 4.4 UX 안정성

### 발견된 문제

- 찜 기능 실패 시 사용자 체감 피드백 약함
- 비밀번호 찾기 패널은 실질 동작이 없어 기대 불일치 가능
- 일부 비동기 기능(찜)에서 로딩/재시도 UX 없음

### 영향

- “버튼 눌렀는데 반응 없음” 인식
- 기능 신뢰도 저하 및 이탈 가능성 증가

### 개선 방향

- 성공/실패 토스트, 버튼 상태 변화(로딩/완료) 도입
- 미구현 기능은 명시적 안내 문구 제공 또는 임시 비활성화
- 네트워크 실패 시 재시도 가이드/행동 유도 추가

## 5. 우선순위별 개선 권장 사항

| 우선순위 | 항목 | 개선 내용 | 예상 효과 |

## 6. 최종 평가

### 전체 평가

- 주요 사용자 플로우(검색/챗봇/상품 상세)는 구조적으로 동작 가능성이 높고, 검색 필터 입력 보정/빈 상태 처리 등 기본 안정성은 양호합니다.
- 다만 보안(챗봇 렌더링 URL), 중복 실행 제어(찜 기능), 실패 피드백 일관성은 우선 보완이 필요합니다.

### 가장 먼저 해결해야 할 문제

- 챗봇 메시지 렌더링의 URL 스킴 검증 부재(보안 우선 이슈)

### 수정 없이 확인 가능한 결론

- 코드 변경 없이 정적 분석 기준으로, 챗봇은 로딩/중복 전송 방지 구현이 상대적으로 우수함
- 검색 필터는 범위 입력값 보정 및 파라미터 정리 로직이 존재함
- 찜 관련 API 연동은 실패/중복 클릭 대응이 상대적으로 약함

### 추가 확인 필요 사항

- 실제 모바일 디바이스에서 레이아웃/터치 인터랙션 검증 (확인 필요)
- 백엔드 응답 포맷 변화(리다이렉트/비JSON) 시 프런트 실패 케이스 재현 테스트 (확인 필요)
- 챗봇 응답 데이터에 악성 링크 포함 시 렌더링/클릭 동작 보안 점검 (확인 필요)
